# v0.0.1-rc21

Canonical tool-call validation, deterministic formal healing, and native-backend compatibility observability.

## Summary

- Enable the canonical runtime tool-call contract by default.
- Enable deterministic formal tool-call healing by default, with an immediate kill switch.
- Add a process-isolated tool-generation comparator and a verified native-backend capability manifest.
- Keep routing, model retry behavior, semantic tool selection, MTP, final-prefix reuse, and tool execution policy unchanged.

## Canonical tool-call validation

All active tool calls pass through one runtime-owned canonical contract before guardrails and execution. The contract requires an exact enabled tool name, object arguments, required fields, exact types, declared ranges, no duplicate keys, and no additional properties. Policy, permission, and operational-limit decisions remain authoritative and are not bypassed.

`ORBIT_TOOL_CALL_CANONICAL_GATE=0` provides an immediate rollback to the legacy validation path. Invalid configuration values disable the gate safely.

## Deterministic formal healing

`ORBIT_TOOL_CALL_HEALING` is enabled by default. Set `ORBIT_TOOL_CALL_HEALING=0` to disable formal healing immediately.

The whitelist is limited to four value-preserving transformations:

- known tool-mode envelope removal;
- trailing-comma removal;
- decoding `arguments` from a string containing one complete JSON object;
- registered-wrapper unwrapping.

Every repaired candidate must preserve the exact tool name, argument keys, types, values, and argument count, then pass the normal canonical contract, policy, permission, operational limits, guardrails, and executor path. The healing module never executes a tool directly.

Ambiguity, multiple candidates, incomplete delimiters or strings, `finish_reason=length`, timeout, cancel, schema failure, policy denial, permission denial, and operational-limit denial remain fail-closed. No aliasing, fuzzy matching, tool substitution, type conversion, defaults, clamps, argument invention/removal/renaming, semantic correction, or model nudge retry is implemented.

## Compatibility observability

The native server exposes a bounded `native_backend_capabilities` manifest for the reviewed Gemma 4 profile. Verified status requires the reviewed `llama.cpp` commit, renderer golden fixtures, tokenizer identity, and exact aligned 64-token final prefix to match.

The generation-only benchmark records versioned corpus, tool-protocol, runtime-configuration, renderer, tokenizer, model, MTP, tools, thinking, affinity, and thread identities without copying prompts or tool arguments into comparison output.

`scripts/compare_tool_call_generation.py` compares baseline and candidate JSONL runs offline. It rejects incomparable fingerprints and fails on semantic regressions, markup leakage, multiple candidates, tool execution, finalization, or additional model calls. Timing is informational only and no speedup is claimed.

## Measured limits

The process-isolated smoke comparison completed 8/8 evaluable scenarios in each process with eight model calls, zero tool executions, and zero finalizations. The observed rates remained visible and unchanged:

- exact-tool match: `0.833333`;
- unwanted attempt: `0.5`;
- budget truncation: `0.125`.

Formal healing does not correct wrong-tool selection, unwanted tool calls, or budget truncation. No semantic-healing claim is made, and synthetic replay alone is not evidence of production utility.

## Existing guards

- Eligible native `final_from_tool` calls retain the exact aligned 64-token prefix reuse enabled in RC20.
- `ORBIT_FINAL_PREFIX_REUSE=0` remains the final-prefix kill switch and restores measured non-reuse behavior with `cached=4`.
- MTP remains authoritative on MTP-owned completions; final-prefix activity remains zero on that path.
- Cancel, timeout, reset, guardrail rejection, and executor errors retain their existing lifecycle behavior.

## Limitations

- Deterministic healing repairs syntax envelopes only; it does not improve semantic tool selection.
- Wrong-tool, unwanted-tool, and budget-truncation events remain distinct and observable.
- The capability manifest is observational and does not gate startup or inference.
- CPU timing depends on workload, output, process state, and thermal conditions; no deterministic performance improvement is claimed.
- The repair whitelist must not be expanded without natural, repeatable malformed production-budget samples and separate safety evidence.
