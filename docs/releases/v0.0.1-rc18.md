# v0.0.1-rc18

Compact final-tool evidence and web-search error correctness release.

## Summary

- Reduce `final_from_tool` prompt tokens while preserving its behavioral contract.
- Reduce compact model-facing evidence metadata while retaining provenance outside the prompt.
- Use a compact, bounded final view for web-search errors.
- Prevent failed web searches from being answered from model memory as if the search succeeded.
- Keep routing, tool-loop, evidence-selection, MTP, cache/KV, and completion-budget architecture unchanged.

## Main changes since RC17

- #128 Use compact final view for web search errors.
- #130 Report web search errors without answering from memory.
- #132 Reduce final from tool prompt tokens.
- #134 Reduce compact evidence prompt metadata.
- #127, #129, #131, #133, and #135 update `AGENTS.md` for the corresponding post-RC17 changes.

## Web-search error handling

#128 routes `web_search` evidence with `status=error` through the compact web
final view. #130 adds an explicit failure marker and narrow instruction so the
model reports the failure briefly instead of answering from general knowledge
as if the failed search had succeeded.

The compact error context retains bounded query, status, and error metadata.
It does not reinject full raw web output. Successful searches, no-result
searches, and non-web errors retain their intended behavior.

## Token reductions

`FINAL_FROM_TOOL_SYSTEM_PROMPT` decreased from 49 to 34 production-tokenizer
tokens. This is an exact deterministic reduction of 15 tokens per
`final_from_tool` call.

Compact model-facing evidence cards now omit audit-only metadata. Full evidence
records, provenance, hashes, raw retrieval, sidecars, and lineage remain
available in EvidenceStore and runtime diagnostics.

Measured compact evidence reductions ranged from approximately 35 to 77 tokens,
depending on the scenario. In the controlled RC17-versus-main comparison, the
comparable `pwd` and shell-error final prompts each used exactly 49 fewer prompt
and evaluated tokens on main.

These token reductions are deterministic. No deterministic wall-time speedup is
claimed; CPU timing observations remain non-authoritative smoke data.

## Cache status

`cached=4` on route-to-final remains unchanged and expected. Route and final
views diverge immediately in their system prompts; this is not a backend cache
failure.

An isolated final-specific checkpoint probe found a reusable 43-token prefix.
Checkpoint restore was bit-exact against an identically segmented baseline, but
the reusable boundary does not align with production prefill segmentation.
Changing that segmentation changes logits, so no production checkpoint reuse or
cache/KV redesign was introduced.

## Warnings / known limitations

- The RC17-versus-main comparison used one E2E run per revision, so it does not support a statistically valid wall-time claim.
- The broader `docs/PROMPTS.md` and ambiguous-prompt E2E matrix was audited but not completed because of CPU cost.
- Live external web behavior was not tested in this comparison; deterministic synthetic web success, no-result, and error paths passed.
- MTP remains optional and experimental and is not guaranteed to improve performance.
- This release does not reopen MTP tuning, evidence selection, final-prefix checkpoint reuse, or route-to-final cache redesign.
