# v0.0.1-rc20

Aligned final-tool prefix reuse release.

## Summary

- Enable aligned 64-token `final_from_tool` prefix reuse by default for eligible native completions.
- Add `ORBIT_FINAL_PREFIX_REUSE=0` as an immediate stable kill switch.
- Preserve legacy `ORBIT_FINAL_PREFIX_EXPERIMENT` compatibility behind stable-variable precedence.
- Validate exact numerical equivalence between cold production prefill, explicit 64-token segmentation, and checkpoint restore.
- Keep routing, tool-loop behavior, evidence selection, MTP behavior, and completion budgets unchanged.

## Why the 43-token experiment was not promoted

RC19 used an optional checkpoint at token 43. The checkpoint implementation was
correct and restore was bit-exact against an identically segmented baseline,
but the boundary did not match production's 64-token prefill segmentation.
Changing that boundary changed Gemma logits and output behavior.

The decisive read fixture returned a concise 13-token stop response through
cold production prefill, but both explicit segmentation at token 43 and
checkpoint restore produced a verbose 96-token length termination. Increasing
the output budget was not an acceptable solution because segmentation, not the
checkpoint or budget, caused the divergence.

## The 64-token solution

The final system instruction is a meaningful 58-content-token policy. Six
Gemma template/control tokens make the complete stable prefix exactly 64 tokens.
It ends at the system turn; the next dynamic token is the user-turn control
token 105. No padding or filler is present.

- Stable-prefix text hash: `c3b8e45ac695a87e60146bb8017a98f1b41fc13a708565b58160cce6d419c6f3`
- Serialized token-prefix hash: `398338fd38a9c80d54b269e09ae70077ab7323ec1a47920879a24896928cdfc5`

Across 11 final families and five repetitions each, cold production prefill,
explicit segmentation at token 64, and checkpoint restore were bit-exact in
55/55 probes. Logits hashes, next token, ordered top-10, bounded output, and
finish reason matched, with maximum logits difference `0.0`.

The former read regression is eliminated: cold, capture, and restore all return
the same 13-token stop response. Checkpoints with the old 43-token identity are
rejected and rebuilt under the 64-token identity.

## Default and kill switch

Eligible native `final_from_tool` completions now use prefix reuse by default.
The first eligible call captures the checkpoint; subsequent eligible calls can
restore `cached=64`.

Set `ORBIT_FINAL_PREFIX_REUSE=0` to disable both runtime payload activation and
native-server reuse immediately. The stable variable overrides
`ORBIT_FINAL_PREFIX_EXPERIMENT`. Legacy-only `0` and `1` configurations remain
compatible. Invalid stable values disable reuse safely and expose a bounded
configuration error without disclosing raw environment values.

Bounded `/props` diagnostics report effective enabled state, configuration
source, validation error, legacy detection, initialized state, prefix tokens,
capture/restore/fallback counters, last-use state, failure reason, and checkpoint
size.

## Results

- Eligible restored finals reach `cached=64`.
- Restore evaluates 60 fewer tokens than the same new prompt without reuse.
- Restore evaluates 36 fewer tokens than previous production.
- The first capture costs 28 additional evaluated tokens.
- Cumulative evaluated-token break-even occurs on the second eligible final.
- The previous 96-token read truncation is replaced by the correct 13-token stop response.
- A targeted high-information matrix passed 40/40 with `finish_reason=stop`.
- Matrix output tokens decreased from 845 to 825 without losing requested facts, paths, counts, matches, errors, or values.

The evaluated-token reduction is deterministic for eligible restored calls.
CPU wall time remains dependent on prompt shape, output length, process order,
system load, and thermal conditions. No deterministic wall-time improvement is
claimed.

## Guards and lifecycle

- MTP remains authoritative; final-prefix capture and restore stay at zero while MTP owns completion.
- Tools-off paths do not activate final-prefix reuse.
- Thinking-enabled final paths remain ineligible.
- Route, chat, tool-call, retry, and repair phases cannot use the checkpoint.
- Prompt mismatch, identity mismatch, capture failure, or restore failure uses the existing safe prefill fallback.
- Failed restore cannot leave the checkpoint initialized.
- Cancel, timeout cleanup, reset, completion error, restart, and context changes invalidate or rebuild state safely.
- Previously recorded lifecycle coverage completed 900 restores and 906/906 correct stop completions with zero normal fallback.
- RSS behavior remained bounded and non-linear, with one retained checkpoint and no stale or cross-process state.

## Known limitations

- Final-prefix reuse applies only to eligible native `final_from_tool` paths.
- The first eligible call captures rather than restores, so benefit begins with later eligible finals.
- CPU timing remains workload- and output-dependent; no general speedup percentage is promised.
- MTP and final-prefix reuse remain mutually exclusive by design.
- Route model-adherence issues and conservative route fallback documented after RC19 remain unchanged.
- This release does not redesign routing, tool planning, evidence selection, or completion budgets.
