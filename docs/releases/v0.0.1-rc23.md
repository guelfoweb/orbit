# v0.0.1-rc23

Native mtmd ABI hardening and verifiable llama.cpp vendor provenance.

## Summary

- Isolate unstable upstream mtmd layouts behind an Orbit-owned native bridge.
- Remove mtmd context, input, capability, and bitmap layouts from Python.
- Record and enforce revision-bound build identity and vendor provenance.
- Keep the production llama.cpp vendor unchanged at `b9551`.

## Stable mtmd ABI Boundary

Python now calls `liborbit-mtmd-bridge` using primitive values and opaque
handles. The bridge constructs `mtmd_context_params` and `mtmd_input_text`
from the headers used by the active native build, initializes all fields used
by Orbit, and adapts the reviewed bitmap return profiles.

The bridge rejects unknown context, input-text, bitmap, and capability ABI
layouts before mmproj initialization. Core ctypes structures that remain in
Python are checked against bridge-reported `sizeof`, `alignof`, and relevant
`offsetof` values. Missing identity, changed artifacts, changed libraries, and
unsupported layouts fail explicitly; there is no silent ABI fallback.

## Revision-Bound Build

The bridge, identity sidecar, and native libraries are co-located. Bridge reuse
is invalidated by changes to:

- bridge source or relevant llama.cpp/mtmd headers;
- native runtime libraries;
- compiler or compiler version;
- CMake configuration or bridge flags;
- upstream source or Orbit patchset provenance.

Native CMake configuration receives explicit `LLAMA_BUILD_COMMIT` and
`LLAMA_BUILD_NUMBER` values from the vendor manifest. They are not derived from
the enclosing Orbit repository.

## Limitations

- The bridge recognizes only reviewed ABI profiles and fails closed on unknown
  layouts. Supporting a future profile requires a separate review and tests.
- Build identity detects local artifact mismatch; it is not a cryptographic
  software-signing mechanism.
- This release makes no performance or speedup claim.
