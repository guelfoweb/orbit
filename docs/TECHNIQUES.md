# Techniques

This document describes the runtime techniques Orbit uses to work well with
Gemma 4 12B through `llama-server`.

The goal is not to replace the model with deterministic task logic. The model
still decides when shell access is needed, which command to run, and how to
write the final answer. The runtime only shapes the contract, bounds expensive
inputs, and keeps long sessions observable.

## Model-driven contract

Orbit uses a small shell-oriented contract:

```text
Answer normally unless shell is needed.
For shell tasks return only:

{"command":"..."}
```

This keeps the model-facing interface simple:

- normal chat remains direct model output
- operational tasks become one shell command request
- the final answer is always generated by the model

The runtime accepts the preferred JSON shape and a few raw `llama.cpp` tool-call
variants when the model emits them. They are normalized into the same internal
shell tool call.

## Tools off by default

Orbit starts with tools disabled.

This reduces both risk and prefill cost. In chat-only mode, Orbit does not send
the shell tool schema and does not enter the tool loop.

The user enables tools explicitly:

```text
/tools on
```

`/tools on` exposes only `exec_shell_full_command`. The terminal renders it as
`exec` for readability, but the internal tool name remains unchanged.

## Separate chat and tools prompts

Orbit uses different system prompts for chat-only turns and tools-enabled
turns.

With tools off, the chat prompt is intentionally short:

```text
Answer normally for conversation, explanation, writing, opinion, and general knowledge.
```

This avoids paying the prefill cost of operational instructions when the user is
only chatting.

With tools on, Orbit uses the longer operational prompt that explains:

- command JSON
- OS and shell
- local files and directories
- web search and explicit URL handling
- analysis evidence requirements
- raw tool-call avoidance

This separation keeps normal chat cheap while preserving command-generation
quality when tools are explicitly enabled.

The prefill estimator follows the same split:

- chat turns are estimated with the short chat prompt
- tools turns are estimated with the operational prompt
- final answers after tool results are estimated with the final-from-tool prompt

## Stable prompt prefixes

`llama-server` can reuse prompt prefixes when they stay stable across turns.

Orbit therefore avoids unnecessary changes to:

- system prompt ordering
- tool schema ordering
- message ordering
- phase prompt structure

Orbit does not rewrite user prompts at ingestion time. Long pasted prompts may
be displayed compactly in the terminal, but the full text is still preserved and
sent to the model.

## Route completeness

The preferred model output for operational tasks is a complete command decision
in the first inference:

```json
{"command":"find . -maxdepth 2 -print"}
```

Avoiding an extra inference often saves more wall time than small token-level
optimizations, especially on CPU-only systems.

Orbit still keeps the decision model-driven. It does not extract paths, URLs, or
commands from the user prompt with deterministic task fast paths.

## Prefill estimation

The `Working (...)` line is a UX estimate, not a backend decision.

Orbit estimates prompt tokens locally and updates an adaptive moving average
from observed backend metrics:

```text
observed_rate = prompt_tokens / prefill_time
```

The estimator is split into lightweight profiles:

- chat
- tool
- final-from-tool

This improves the displayed estimate without using `cached_tokens`, which is
not reliable enough as a predictor of perceived latency.

No routing, memory, or tool decision depends on this estimate.

## Tool result reinjection

Tool results are passed back to the model for the final answer. The terminal
makes this visible:

```text
exec {"command": "orbit-web-search 'Dante Alighieri'"}
 └ 2299 chars -> model
```

The `-> model` suffix means the result is being reinjected into the next model
inference.

Large reinjected results are one of the main sources of CPU-only latency. Orbit
does not hide this cost.

## Bounded shell output

Shell execution is bounded by timeout and output-size limits.

The runtime does not synthesize shell output locally. It may only:

- bound output size
- structure output
- clean transport noise such as HTML tags
- reject unsafe or malformed tool calls

The final interpretation remains model-generated.

## Long file handling

Small UTF-8 text/source files can be returned directly.

Large `cat` output is converted into the same bounded reader shape used by the
internal file reader:

```text
shell_output_read_file: true
chunk_index: 0
total_chunks: 3
content:
...
```

This gives the model real content without flooding the context window.

Chunking limits are intentionally bounded:

- complete read up to 256 KB
- chunk mode up to 1 MB
- 6k chars default chunk size
- 12k chars max chunk size
- max 3 chunk reads per user turn

## PDF text extraction

PDFs are not passed to the model as raw binary data.

For PDF reading or analysis, Orbit prefers:

```text
pdftotext
```

If `pdftotext` is not available, it falls back to filtered `strings`.

There is no OCR and no Python PDF dependency. Extracted text follows the same
bounded/chunked reinjection discipline used for long text files.

## Web search and URL handling

Generic web search uses the internal shell command:

```text
orbit-web-search "query"
```

It fetches DuckDuckGo HTML search results with a browser-like User-Agent, parses
the result page locally, and returns bounded structured evidence:

```text
web_search_results: true
query: Dante Alighieri
results:
1. title: ...
   url: ...
   snippet: ...
```

The model receives title, URL, and snippet. It does not receive raw DuckDuckGo
HTML.

Explicit URLs use shell commands such as `curl`.

When `curl` returns HTML, Orbit converts it to readable text before reinjection
unless the user explicitly asks for HTML/page source analysis. In that case,
source-like HTML is preserved.

## Final answer policy

Final answers are always generated by the model.

Orbit uses compact final instructions after tool results to reduce unnecessary
output:

- web search: concise bullets, expand only if asked
- shell output: compact findings, preserve numbers and names
- file reads: respect requested length; otherwise answer concisely
- lists: return names compactly

This reduces generation time without locally summarizing the answer.

## Output truncation

If the backend returns `finish_reason=length`, Orbit does not auto-continue.

It shows the partial answer and exposes explicit recovery commands:

```text
/continue
/max-tokens N
```

This avoids hidden extra inference cost and keeps control with the user.

## Guardrails

Guardrails are runtime boundaries that keep the model-driven workflow safe and
recoverable. They do not decide the task answer and they do not replace the
model's reasoning. They enforce contracts, prevent pathological loops, bound
inputs, and ask the model for correction when enough evidence is available.

The important design rule is: **guide the model, do not solve the task in
runtime code**.

When a guard fires, Orbit gives the model a smaller, clearer internal task:
produce corrected JSON, inspect real content, verify a mutation, or continue a
requested edit. The runtime does not infer the correct patch, choose the target
file, validate domain-specific semantics, or synthesize the final answer.

Orbit groups guardrails into a few categories:

- contract guardrails: valid tool-call shape, JSON recovery, unknown tool rejection
- loop guardrails: bounded rounds, repeated tool-call detection, retry budgets
- execution guardrails: timeouts, output-size limits, workdir confinement
- evidence guardrails: content-evidence recovery, metadata-only retry, mutation verification
- task-completion guardrails: completion nudge, minimal patch nudge, semantic repair
- context guardrails: bounded chunks, memory refresh, manual tool-result compaction

### Loop and contract guardrails

Orbit keeps several runtime invariants in code:

- bounded tool rounds, with a slightly larger shell-full budget for repair workflows
- repeated tool-call detection
- timeout enforcement
- malformed JSON/tool-call recovery
- unknown tool rejection
- metadata-only analysis retry for tools-on evidence requests

These guardrails do not decide the task answer. They prevent pathological loops
and invalid states.

### Dynamic tool-call budget

Route/tool-call turns are not final answers and should not use the same output
budget.

Orbit keeps ordinary tool-call turns small, but allows a larger budget for
mutative/internal correction turns:

- normal tool-call: compact JSON only
- mutative tool-call: larger budget for safer edit commands
- shell repair: larger budget for corrected commands
- mutation verification: enough budget for evidence-producing commands
- completion/minimal-patch/content-evidence guards: enough budget for one
  follow-up command

This reduces `finish_reason=length` in internal turns without increasing the
cost of simple commands.

The model still generates the command. The runtime only chooses an output budget
appropriate to the phase.

### Shell repair loop

Shell commands generated by the model can fail because of syntax, quoting,
regexes, missing paths, unsupported options, or differences between local
utilities.

Orbit handles this with a model-driven repair loop:

```text
exec command
 -> exit code non-zero
 -> inject exit code, stdout, stderr
 -> ask the model for corrected JSON
 -> execute the corrected command
```

The internal repair prompt is intentionally narrow:

```text
The previous shell command failed.

Exit code: <code>

STDOUT:
...

STDERR:
...

Return only corrected JSON:

{"command":"..."}
```

The runtime does not rewrite or fix commands itself. It only exposes the real
failure evidence and lets the model choose the next command.

Limits:

- only `exec_shell_full_command` is eligible
- maximum 2 repair retries
- repeated identical tool calls are rejected by the tool-loop state
- stdout and stderr reinjected into the repair prompt are bounded

Most non-zero exits are treated as potentially repairable. Orbit skips repair
only for clearly environmental failures such as permission errors, read-only
filesystems, no space left, out-of-memory/killed processes, DNS failures, and
network timeouts.

### Content evidence recovery

Coding and analysis tasks fail when the model stays at the metadata level:

```text
ls -R
ls -F
file target.py
```

Listing files can be useful, but it is not enough to fix code or analyze a
sample. When a mutative coding task triggers metadata-only rejection and no
source/test content has been read yet, Orbit asks the model for real evidence:

```text
Your previous command inspected only metadata or listings.

For this coding task, inspect real file or test content before continuing.

Use commands such as cat, sed -n, grep/rg on file contents, or test output.

If file names are unknown, use grep/rg recursively over file contents.

Do not use ls, find, tree, file, or stat.

Return only JSON:

{"command":"..."}
```

The runtime does not choose the file to read. It only prevents another
listing-only step and asks the model to inspect content. After that, the normal
flow continues:

```text
content evidence -> modification -> verification -> final answer
```

Visible counters:

- `content_evidence_guard_nudges`
- `content_evidence_guard_commands`
- `content_evidence_guard_successes`
- `content_evidence_guard_failures`

### Mutation verification

Some mutating commands exit successfully but do not actually perform the
requested change. Examples include:

- a `sed -i` expression that matches nothing
- `mv` or `cp` operating on the wrong target
- a database update that touches zero rows
- a create/write command hidden behind `|| true`

These are not shell errors, so the shell repair loop cannot detect them.

Orbit therefore has a separate model-driven mutation verification step. If a
shell command:

- exits with status 0
- returns no stdout/stderr
- appears mutating
- and the user request is not clearly read-only

the runtime asks the model for one verification command:

```text
The command succeeded but produced no output.

Verify that the requested change actually occurred.

The verification command must print direct evidence of the changed value or state, not only metadata.

Return only JSON:

{"command":"..."}
```

Again, the runtime does not validate HTML, SQL, YAML, Git state, or any
domain-specific format. The model chooses the verification command and later
generates the final answer from the available evidence.

Limits:

- maximum 1 mutation verification per tool turn
- maximum 1 repair after a failed verification
- no semantic validation in runtime
- no format-specific success checks

Mutation verification counters are visible in `/status`:

- `mutation_verifications`
- `mutation_verification_repairs`
- `mutation_verification_failures`

### Completion guard

Some mutative coding tasks were correctly discovered but never edited. The model
read the right file, identified the target, then finalized as if the task were a
review.

Orbit detects this pattern only when all of these are true:

- the user request is mutative
- shell-full is enabled
- only read-only commands were executed
- no mutating command was attempted
- the model is about to finalize

The internal nudge is deliberately small:

```text
You identified the target but did not perform the requested modification.

Continue the task.

Prefer short robust commands; avoid fragile quoting and long heredocs.
Prefer minimal edits over rewriting entire files.

Return only JSON:

{"command":"..."}
```

The runtime does not suggest the edit. It only prevents premature completion.

Visible counters:

- `completion_guard_nudges`
- `completion_guard_commands`
- `completion_guard_successes`
- `completion_guard_failures`

### Minimal patch guard

The model can understand the requested change but attempt to rewrite an entire
file through a long heredoc or a very large command. On CPU-only local models,
this often causes:

- `finish_reason=length`
- incomplete JSON
- truncated heredocs
- fragile full-file rewrites

When the task is mutative, a target has been discovered, a broad rewrite is
detected, and no valid mutation has succeeded yet, Orbit asks for a local patch:

```text
Your previous command tried to rewrite too much and was too long or incomplete.

Use a minimal local patch to modify only the necessary lines.

Do not rewrite the whole file unless strictly necessary.

Return only JSON:

{"command":"..."}
```

Again, the runtime does not generate the patch. It only redirects the model away
from an overly broad strategy.

Visible counters:

- `minimal_patch_guard_nudges`
- `minimal_patch_guard_commands`
- `minimal_patch_guard_successes`
- `minimal_patch_guard_failures`

### Semantic mutation repair

Verification can show that a mutation was applied but did not satisfy the full
request. For example, a command may change one required property but miss
another, or a test command may print `False`.

Orbit can ask for one follow-up command:

```text
The previous modification was applied, and the verification output is in context.

Check whether all requested changes are satisfied.

If any requested change is missing, return a minimal follow-up command to complete it.

If all requested changes are satisfied, return only: OK

Return command JSON only when a follow-up command is needed:

{"command":"..."}
```

This is still model-driven. The runtime does not know what counts as correct
HTML, Python, shell, SQL, or JSON. It only gives the model verification evidence
and one chance to complete the task.

Visible counters:

- `mutation_semantic_repairs`
- `mutation_semantic_repair_commands`
- `mutation_semantic_repair_failures`

## Context pressure

Orbit reports estimated context pressure in the footer and `/status ctx`.

CPU-only sessions can slow down well before the context window is full,
especially when old tool results dominate the active context.

Typical pressure labels:

```text
ctx: 4280/8192 (52%) | pressure: moderate
ctx: 5900/8192 (72%) | pressure: high
```

## Session memory refresh

Orbit can refresh long-session memory using the model.

The refresh is internal and not stored as a visible user turn. It rebuilds active
history from:

- system prompt
- model-generated durable memory
- recent verbatim tail

The refresh is discarded if it is empty, attempts tools, fails, or does not
reduce estimated context.

Memory state is visible in `/status`.

## Tool result compaction

Old heavy tool results can be compacted manually:

```text
/compact tools
```

The compaction is model-generated. The persisted session remains complete; only
the active context representation is replaced.

The command reports:

- candidates
- tool involved
- before/after token estimate
- tokens saved
- age of the tool result

Automatic tool-result compaction is intentionally not enabled.

## Streaming and terminal feedback

Orbit streams final answers when the backend supports streaming.

Before the first token, it shows a compact elapsed-time indicator with estimated
prefill progress. Tool events are shown in dim text and kept short:

```text
exec {"command": "ls -F"}
 └ 50 chars -> model
```

This keeps CPU-only latency visible without turning the CLI into a full-screen
TUI.

## Backend profile

The default server profile uses Gemma 4 12B with MTP speculative decoding.

MTP improves generation throughput, while most prefill cost still comes from
prompt size, context, tool schemas, and reinjected tool results.

See [Performance notes](PERFORMANCE.md) for benchmark data and server-profile
details.
