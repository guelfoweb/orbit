# Techniques

This document summarizes the runtime techniques Orbit uses with its current local backends, primarily native `orbit server`.

The goal is not to replace the model with deterministic task logic. The model still decides when shell access is needed, which command to run, and how to write the final answer. The runtime shapes the contract, bounds expensive inputs, and keeps long sessions observable.

## Model-driven contract

Orbit uses a small shell-oriented contract:

```text
Answer normally unless shell is needed.
For shell tasks return only:

{"command":"..."}
```

This keeps the model-facing interface simple:

- normal chat remains direct model output
- operational tasks become one shell command request
- the final answer is always generated by the model

The runtime accepts the preferred JSON shape and normalizes backend-specific tool-call variants into the same internal shell tool call.

## Tools off by default

Orbit starts with tools disabled.

This reduces both risk and prefill cost. In chat-only mode, Orbit does not send the shell tool schema and does not enter the tool loop.

The user enables tools explicitly:

```text
/tools on
```

`/tools on` exposes only `exec_shell_full_command`.

## Separate chat and tools prompts

Orbit uses different system prompts for chat-only turns and tools-enabled turns.

This keeps normal chat cheaper while preserving command-generation quality when tools are explicitly enabled.

## Stable prompt prefixes

The backend can reuse prompt prefixes when they stay stable across turns.

Orbit therefore avoids unnecessary changes to:

- system prompt ordering
- tool schema ordering
- message ordering
- phase prompt structure

Orbit does not rewrite user prompts at ingestion time.

## Prefill estimation

The `Working (...)` line is a UX estimate, not a backend decision.

Orbit estimates prompt tokens locally and updates an adaptive moving average from observed backend metrics.

No routing, memory, or tool decision depends on this estimate.

## Tool result reinjection

Tool results are passed back to the model for the final answer. Large reinjected results are one of the main sources of CPU-only latency, so Orbit bounds them instead of hiding the cost.

## Bounded shell output

Shell execution is bounded by timeout and output-size limits.

The runtime does not synthesize shell output locally. It may only:

- bound output size
- structure output
- clean transport noise such as HTML tags
- reject malformed tool calls

Interpretation stays model-generated.

## Long file handling

Small UTF-8 text or source files can be returned directly.

Large reads are converted into bounded chunks so the model sees real content without flooding the context window.

## PDF text extraction

For PDF reading or analysis, Orbit prefers `pdftotext`. If unavailable, it falls back to filtered `strings`.

There is no OCR and no raw PDF reinjection.

## Web search and URL handling

Generic web search uses:

```text
orbit-web-search "query"
```

Explicit URLs should be fetched with `curl`.

HTML may be converted into readable text unless the user explicitly asks for source analysis.

## Session memory

Orbit uses model-driven session memory refresh for long sessions.

The model writes the memory. The runtime only controls when refresh is attempted, whether it actually shrinks context, and how rebuilt history is assembled.

## Guardrails

Orbit keeps guardrails generic:

- content evidence guard
- shell repair loop
- mutation verification
- completion guard
- minimal patch guard

These guardrails guide the model. They do not solve the task deterministically.

## Native backend notes

The runtime owns behavior.

The backend owns:

- tokenization
- KV/session state
- decoding
- streaming transport
- cancellation
- backend metrics

Orbit should keep that boundary explicit.
