# KV Cache Reuse Plan

Status: planning only. This document defines the analysis and benchmark work required before any KV cache reuse patch.

## Goals

Any KV cache reuse work must be justified by measurable local improvements without changing correctness.

Primary metrics:

- lower prompt prefill tokens evaluated
- lower time to first visible token
- lower wall time on repeated or multi-turn prompts
- stable or improved cache reuse metrics in the footer
- no regression in final-answer correctness
- no stale context or cross-session contamination

## Stable Prompt Prefix Candidates

Potentially reusable prompt prefix sections:

- base system prompt
- runtime policy text
- tool definitions when tool mode and tool set are unchanged
- local capability summary while capabilities have not been refreshed
- stable session prefix before recent conversation tail
- memory summary generated by the model, until the next memory refresh

These sections are reusable only if their byte/token representation is identical.

## Frequently Changing Prompt Sections

Prompt sections that commonly invalidate reuse:

- current user message
- recent assistant responses
- tool calls and tool results
- dynamic evidence from files, URLs, directories, and system tools
- refreshed capability summaries
- tools on/off changes
- thinking on/off changes
- media attachments
- memory refresh output
- any timestamp, elapsed-time, or diagnostic text if included in model context
- max-token or phase-specific instructions when they alter prompt content

KV reuse must not assume these sections are stable.

## Existing Reuse To Measure First

Before adding new reuse logic, measure what already exists:

- native server session reuse
- backend KV cache behavior for repeated prefixes
- prompt cache metrics in the footer
- `cache_prompt` behavior, if exposed by the active backend path
- repeated-turn behavior within one Orbit session
- behavior after `/reset`
- behavior when switching `/tools on` and `/tools off`

The first patch in this area should be diagnostics or benchmark capture, not behavior change.

## Baseline Benchmark Matrix

Run these baselines before any KV patch:

| Case | Purpose |
| ---- | ------- |
| chat simple, tools off | baseline chat prefill and TTFT |
| tools off multi-turn | repeated prompt-prefix reuse without tool schemas |
| tools on, no tool needed | static tool prompt overhead |
| file-read README | tool result reinjection with content evidence |
| `fetch_url` direct URL | structured URL evidence payload |
| `list_directory` workdir | compact directory evidence |
| `system_info` specs | compact system evidence |
| long prompt | large user input behavior |
| repeated same-session turn | cache reuse in a stable session |
| `/reset` then prompt | reset invalidation behavior |
| tools on/off switching | tool schema and prompt invalidation |

Metrics to record:

- total context tokens
- estimated prefill tokens
- backend evaluated tokens
- cached tokens
- cache percentage
- prefill tokens per second
- generation tokens per second
- time to first visible token
- total wall time
- finish reason
- correctness outcome

Use `orbit bench-core` for public regression coverage and add temporary local scripts only for deeper profiling.

For phase-1 cache diagnostics, use:

```bash
ORBIT_KV_DIAG=1 python3 scripts/bench_kv_diag.py
```

The script writes raw diagnostic JSONL and a compact Markdown summary under `benchmarks/`. Benchmark outputs are local artifacts and should not be committed unless explicitly requested.

## Invalidation Risks

KV reuse risks:

- stale tool definitions after tool mode changes
- stale capability summary after `/tools refresh`
- stale memory summary after session memory refresh
- stale prompt prefix after `/reset`
- cross-session contamination between workdirs or sessions
- mixing thinking-on and thinking-off prompts
- mixing tools-on and tools-off prompts
- reusing cache after dynamic tool evidence changes
- hidden prompt drift from config, system prompt, or backend options
- backend-specific cache semantics differing between native `orbit server` and OpenAI-compatible backends

Correct invalidation is more important than small speedups.

## Design Questions

Questions to answer before implementation:

- What exact prompt prefix can be hashed safely?
- Should reuse be scoped to one backend session, one Orbit session, one workdir, or one process?
- Which runtime states must be part of the cache key?
- Can the native server expose enough prefix/KV diagnostics to validate reuse?
- How should `/reset`, `/tools on`, `/tools off`, `/think on`, `/think off`, `/tools refresh`, and memory refresh invalidate reuse?
- What is the fallback behavior when the backend cannot reuse KV safely?
- How can tests detect stale context without becoming prompt-specific?

## Experiment Protocol

1. Capture baseline metrics with no behavior changes.
2. Add diagnostics only if current metrics are insufficient.
3. Prototype reuse behind an explicit experimental flag.
4. Compare against the same model, backend, context, workdir, and prompts.
5. Validate correctness before considering latency gains.
6. Keep rollback trivial if any stale-context behavior appears.

## Acceptance Criteria

A KV cache reuse patch is acceptable only if:

- it shows repeatable reduction in prefill or TTFT on benchmarked cases
- it does not alter model-facing task semantics
- it does not change final-answer policy
- it does not bypass evidence guards
- it correctly invalidates on reset, tool mode, thinking mode, capability refresh, memory refresh, and dynamic evidence changes
- it cannot contaminate across sessions or workdirs

## Out Of Scope

- MTP changes.
- Browser automation.
- Deterministic task routing.
- Prompt rewriting to fake reuse.
- Backend/native behavior changes before baseline measurements justify them.
